## 3. Замер критики (угодливость)

5 битых наборов в `measure_critic.py` (`FAKE_BROKEN`). Скрипт готов:

```bash
python measure_critic.py -n 10          # все кейсы
python measure_critic.py --case 0 -n 10 # по одному
```

**Разовая проверка** (кейс «арифметика без calculate»): при T=0.0 и T=0.7 Критик оба раза поставил `ok=True` — **ложное принятие** (ожидаемо для битого ответа без `calculate` в `used_tools`).

Полная таблица N=10 не собрана (прервано 429). Ожидаемый формат:

| Битый кейс | T=0.0 | T=0.7 |
|---|---:|---:|
| арифметика без calculate | —/10 | —/10 |
| выдуманное число | | |
| несогласованные данные | | |
| ответ с ошибкой исполнителя | | |
| план не покрывает вопрос | | |

**Гипотеза:** при T=0.0 Критик чаще зеркалит план и принимает битые ответы; T=0.7 даёт шум, но на явных нарушениях (нет `calculate`, `(ошибка: ...)`) должен быть строже — полная проверка требует прогона `measure_critic.py -n 10`.

## 5. Вывод: когда PWC оправдан

**PWC оправдан**, когда:
- нужна декомпозиция с `depends_on` и параллель на уровне (Q1: ускорение 1.71×);
- Планировщик склонен к выдуманным tools — **валидатор схемы обязателен** (Q3);
- задача multi-hop (реальная ставка, кросс-курс).

**Одиночный агент лучше**, когда:
- 1–2 tool calls, overhead PWC (план + N workers + критик + синтез) не окупается;
- накопленная инфляция за 50+ месяцев — честный план слишком тяжёл для Исполнителя.

**Самый хрупкий элемент** — согласованность Планировщика с доступными tools и лагом данных; без валидатора галлюцинации в плане не чинятся. Критик при T=0.0 склонен к ложному принятию.
