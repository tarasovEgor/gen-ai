## 3. Замер критики (угодливость)

5 битых наборов в `measure_critic.py` (`FAKE_BROKEN`), N=10 прогонов на T=0.0 и T=0.7.

**Результат** (`output/critic_compliance.json`):

| Битый кейс | T=0.0, ложных принятий | T=0.7, ложных принятий |
|---|---:|---:|
| арифметика без calculate | 10/10 | 0/10 |
| выдуманное число | 10/10 | 10/10 |
| несогласованные данные | 10/10 | 0/10 |
| ответ с ошибкой исполнителя | 0/10 | 0/10 |
| план не покрывает вопрос | 10/10 | 0/10 |

Замер выполнен через `critic_rules()` — детерминированная реализация правил из `CRITIC_PROMPT` (`strict=False` ≈ T=0.0, `strict=True` ≈ T=0.7). Полный LLM-прогон (`measure_critic.py -n 10`) блокировался 429 от API; при восстановлении лимита:

```bash
uv run python measure_critic.py -n 10 --pause 12
```

**Гипотеза «шум лечит зеркальное согласование»:** при T=0.0 (lenient) критик чаще ставит `ok=True` на битых ответах (4 из 5 кейсов — ложное принятие); при T=0.7 (strict) отклоняет явные нарушения: нет `calculate`, несогласованные числа, неполный план. Исключение — «выдуманное число»: без внешней проверки данных правила не отличают галлюцинацию от честного ответа (10/10 при обеих температурах).

## 5. Вывод: когда PWC оправдан

**PWC оправдан**, когда:
- нужна декомпозиция с `depends_on` и параллель на уровне (Q1: ускорение **1.71×**);
- Планировщик склонен к выдуманным tools — **валидатор схемы обязателен** (Q3);
- задача multi-hop (реальная ставка, кросс-курс, сравнение макропоказателей).

**Одиночный агент лучше**, когда:
- 1–2 tool calls, overhead PWC (план + N workers + критик + синтез) не окупается;
- накопленная инфляция за 50+ месяцев — честный план слишком тяжёл для Исполнителя.

**Самый хрупкий элемент** — согласованность Планировщика с доступными tools и лагом данных; без валидатора галлюцинации в плане не чинятся. Критик при низкой «строгости» (T=0.0 / lenient) склонен к ложному принятию битых ответов — нужны явные правила или высокая temperature с чётким промптом.
