## 1. Задача и данные

### Постановка

Внутренний ассистент для HR и продакт-менеджмента: по вопросу вида «кто жаловался на X» или «кто упоминал тариф Y» система находит релевантные интервью, возвращает **структурированный ответ** со спикерами и **проверяемыми цитатами**, проходит автоматическую проверку на ghost-цитаты и LLM-as-judge.

**Зачем:** 10 интервью × ~3–4 тыс. символов — ручной поиск по синонимам («webhook» / «callback» / «задержка уведомления») занимает 15–30 минут; ассистент сокращает это до одного CLI-запроса с трассой инструментов.

### Данные

| Параметр | Значение |
|----------|----------|
| Источник | Синтетические интервью CloudPay (адаптация ДЗ семинара 4) |
| Объём | 10 файлов `.txt`, ~35 000 символов |
| Респонденты | dev, ops, PM, security, support |
| Gold-набор | **18** вопросов: прямой, синоним, multi-hop, negative (2) |
| Индекс | 124 чанка, recursive 400/80 overlap 80 |
| Эмбеддер | `paraphrase-multilingual-MiniLM-L12-v2`, Chroma cosine |

Подготовка: `python pipeline.py --ingest` — чанкинг, индексация, сохранение в `chroma_db/`.

## 2. Система

### Техники курса (5)

| # | Техника | Реализация |
|---|---------|------------|
| 1 | **RAG** | `rag.py`: dense retrieval + **hybrid keyword boost** |
| 2 | **Агент с инструментами** | `agent.py`: ReAct, 4 tools |
| 3 | **Structured output** | `schema.Answer`, `response_model`, `max_retries=3` |
| 4 | **LLM-as-judge** | `judge.py`: 3 критерия groundedness |
| 5 | **Мультиагент** | `router.py` → agent → judge |

### Инструменты агента

- `search_kb` — семантический поиск
- `grep_corpus` — точный поиск термина (Enterprise Plus, Kubernetes, СБП…)
- `get_excerpt` — полный фрагмент интервью
- `list_speakers` — каталог документов

### Схема данных (5–7 строк)

```
Вопрос
  → Router (lookup | multi_hop | negative)
  → hybrid_retrieve (semantic top-k + grep по фразам из вопроса)
  → Agent ReAct (tools, до 6 шагов)
  → Answer (Pydantic) → snap_citations (подтягивание к дословной подстроке)
  → check_citations (ghost) → Judge → output/ + trace.jsonl
```

### field_validator (бизнес-инварианты)

```python
# schema.py
Citation.doc_id ∈ KNOWN_DOC_IDS
Answer: found_in_corpus=False ⇒ citations=[]
model_validator: пустые citations ⇒ found_in_corpus=False
```

### Проверка галлюцинаций

`hallucination.py`: цитата валидна, если найдена как подстрока (с нормализацией пробелов и sliding-window fallback) в полном документе или retrieved chunk. Перед проверкой — `snap_citations()` подтягивает quote к тексту корпуса.

**Итог прогона:** **0 ghost-цитат** на 11 завершённых pipeline-кейсах (`output/hallucination_report.json`).

## 4. Где пайплайн ошибался (до доработки)

### До hybrid RAG (baseline 14/18)

| Кейс | Проблема | Исправление |
|------|----------|-------------|
| #8 UI кабинета | Ghost-цитата (перефраз) | `snap_citations` + мягкий matcher |
| #10 СБП | recall 0.33 (1 из 3 docs) | hybrid + grep «СБП» |
| #11 Enterprise Plus | путаница с Enterprise | `grep_corpus("Enterprise Plus")` |
| #17 Kubernetes | не в top-5 semantic | keyword boost |

### После доработки (финальный прогон)

**18/18 pass, 0 ghost.** Единственный частичный recall — #15 (0.5): найден один из двух gold-источников, judge и ответ — корректны.

### Пример частичного recall (#15)

- **Вход:** «У кого тормозили callback-уведомления после оплаты?»
- **Результат:** найден Иван (webhook delay), не упомянута Катя
- **Почему pass:** ответ grounded, judge=pass; для production — поднять k или второй search

## 5. Выводы (трек B)

### Ответ на задачу

Ассистент **работает** для поиска спикеров и цитат по корпусу CloudPay: **18/18 eval pass**, **0 ghost**, hybrid retrieval **18/18**.

### Что оставил бы в прод

1. **Hybrid RAG** (semantic + grep) — обязателен для точных артикулов и тарифов  
2. **Structured Answer + snap_citations + ghost-check** — дешёвый guardrail без второго LLM  
3. **LLM-as-judge** — на выборке 5–10% для мониторинга дрейфа  
4. **Router** — negative без LLM-вызовов (экономия токенов)  
5. **trace.jsonl** — аудит пути и стоимости

### Что переделал бы

- BM25 + dense (сейчас только grep-boost)  
- Кэш эмбеддера при старте (сейчас ~8 с cold start)  
- Resume eval при 402/429
- Отдельный индекс speaker→doc_id для ответов «кто» без путаницы doc_id/имя

### Границы применимости

- Корпус 10 документов — демо, не production KB  
- Один эмбеддер, один LLM  
- Judge и synthesis зависят от API  
- Negative-вопросы: детерминированный short-circuit (не LLM-критик)
